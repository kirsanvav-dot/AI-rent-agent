## 1. Бизнес-контекст и цели

### 1.1 Проблема

Владелец квартиры для посуточной аренды вынужден:

- Вручную отвечать клиентам в Telegram, Авито и других каналах (1–2 часа 3 раза в неделю).
- Синхронизировать календари вручную: бронь с одной площадки нередко конфликтует с другой.
- Платить комиссию агрегаторам (Booking.com, Суточно.ру) за каждую бронь.
- Терять данные о гостях из-за отсутствия времени на аналитику.
- Терять данные о чистой прибыли с квартиры из-за отсутствия времени на аналитику.

### 1.2 Решение

Единый **AI-агент** (Express), который:

1. Принимает вебхуки от площадок в одной точке входа (уже работает — см. `AS_IS.md` §1).
2. Ведёт **единую CRM** в Notion для всех броней и лидов.
3. **Двунаправленно** синхронизирует календари: RealtyCalendar → Авито (уже работает) и Авито/прямая бронь → ЦИАН/Яндекс **через RealtyCalendar**.
4. Отвечает гостям в Telegram и Авито от имени «Анны» **с контекстом брони**, когда гость идентифицирован.
5. Даёт владельцу управление статусами брони из Telegram без ручного захода в Notion.

### 1.3 Бизнес-цели и метрики

| ID | Цель | Метрика успеха |
|----|------|----------------|
| BG-1 | Сократить время ручной работы на 80 % | ≤ 10 мин/день ручного труда к концу 1-го месяца после выката |
| BG-2 | Устранить двойные бронирования | 0 конфликтов дат в течение 30 дней |
| BG-3 | Снизить зависимость от агрегаторов | Доля прямых броней (Авито / Яндекс / ЦИАН / Telegram) ≥ 60 % за квартал |
| BG-4 | Единая CRM для всех броней | 100 % подтверждённых броней и лидов из Авито в Notion в режиме реального времени |
| BG-5 | Отвечать клиентам быстрее | Время первого ответа ≤ 60 секунд 24/7 |

### 1.4 Принятые продуктовые решения (из GAP_ANALYSIS)

| GAP | Решение | Влияние на скоуп |
|-----|---------|------------------|

| GAP-003 / GAP-012 | Публичный URL вебхуков **ещё не принимает реальные брони** (создание URL в процессе) | Верификация подписей — **вне текущего скоупа** (§6) |

### 2.2 Изменённые и новые сценарии

Для каждого указано, **что заменяет в AS-IS**.

#### US-CHG-1 — Идентификация гостя в Telegram

**Заменяет в AS-IS:** поиск `findBookingByChatId` всегда даёт `null`, т.к. поле `Telegram chat_id` никогда не заполняется; каждый гость с существующей бронью обрабатывается как «незнакомый» (`AS_IS.md` §1, §4.1, GAP-002).

**Актор:** гость с уже созданной бронью (RealtyCalendar / другой источник).

**Поток (TO-BE):**

1. Гость пишет боту в Telegram.
2. Агент ищет бронь по `Telegram chat_id`.
3. Если не найдено — агент предлагает **привязку**:
   - **Вариант A:** гость отправляет **код брони** (из уведомления / переписки; формат: значение поля `ID Брони` в Notion).
   - **Вариант B:** гость отправляет **контакт** (кнопка «Поделиться номером» / `message.contact`).
4. При успешном матчинге агент записывает `Telegram chat_id` в карточку Notion.
5. Дальнейшие сообщения обрабатываются с полным контекстом брони (имя, квартира, даты, сумма, источник).

**Критерий приёмки:** после привязки второе сообщение того же `chat_id` находит бронь без повторного ввода кода/телефона.

#### US-CHG-3 — Реальная интеграция Авито: Messenger ↔ CRM

**Заменяет в AS-IS:**

- US-OK-5 (базовый Авито без CRM) — в production Авито-роут будет **всегда** работать с CRM.
- US-OK-2 (демо-режим как основной режим) — `DEMO_MODE` больше не используется в production (см. NFR-10).
- `AS_IS.md` §1 п.3, §3.4, GAP-006, GAP-007: Авито-роут не пишет в Notion, контекст LLM всегда пуст, `Авито chat_id` не заполняется.

**Акторы:** владелец (один раз настраивает интеграцию), гость в Messenger Авито (рантайм).

**Предусловия (одноразовая настройка владельцем):**

1. Получены реальные `AVITO_CLIENT_ID`, `AVITO_CLIENT_SECRET`, `AVITO_USER_ID`.
2. **Webhook URL зарегистрирован в кабинете Авито** (или через API подписки) — события Messenger о новых сообщениях идут на `POST <PUBLIC_URL>/webhook/avito`. См. FR-18.
3. `DEMO_MODE` отсутствует или `false`.

**Поток (TO-BE) — рантайм:**

1. Гость пишет в чат объявления → Авито шлёт вебхук на `POST /webhook/avito`.
2. Агент валидирует событие (`payload.type === 'message'`, автор ≠ `AVITO_USER_ID`).
3. Агент получает OAuth2-токен Авито (с in-memory кешем — поведение AS-IS сохраняется).
4. Агент **ищет запись в Notion** по `Авито chat_id` (новый метод `findBookingByAvitoChatId`).
5. Если запись **не** найдена — создаёт **черновую бронь** (лид) в Notion:
   - `Статус` = `Ожидает подтверждения`;
   - `Источник` = `Авито`;
   - `Авито chat_id` = ID чата;
   - `Авито item_id` = ID объявления (если приходит в payload);
   - `Имя клиента` / `Телефон` / даты — заполняется из payload, если доступно, иначе пусто (агент уточнит у гостя в LLM-диалоге);
   - `ID Брони` = `AVITO-{chatId}` (стабильный ID для дедупликации).
6. Если запись **найдена** — обновляет нужные поля (например, имя/телефон, если гость их назвал).
7. LLM-вызов получает контекст из найденной/созданной записи (FR-14).
8. Ответ отправляется через Messenger API Авито.
9. Если гость согласовал даты — владелец видит уведомление в Telegram с кнопкой `Подтвердить бронь` (триггер US-CHG-6, блокировка через RC).

**Критерии приёмки:**

- При первом сообщении в новом Авито-чате в Notion появляется ровно одна запись со статусом «лид».
- Повторные сообщения в том же чате **не** создают дубликат и используют тот же контекст.
- В production-режиме (`DEMO_MODE=false`) при отсутствии валидных Авито-ключей агент **падает на старте** или явно логирует ошибку конфигурации, а не молча работает в demo.
- Лид из Авито виден владельцу в Notion наравне с подтверждёнными бронями (BG-4).

#### US-CHG-4 — Владелец меняет статус брони из Telegram

**Заменяет в AS-IS:** `updateBookingStatus` не вызывается в production; все брони навсегда `Подтверждена`; смена только вручную в Notion (GAP-004).

**Актор:** владелец.

**Поток (TO-BE):**

1. При уведомлении о новой брони (или по команде «мои брони» — опционально в бэклоге) владелец видит inline-кнопки, например: `Заехал` | `Завершена` | `Отменена`.
2. Нажатие → `callback_query` → обновление поля `Статус` в Notion → подтверждение владельцу в Telegram.

**Жизненный цикл статусов (минимум):**

`Подтверждена` → `Заехал` (активная аренда) → `Завершена`  
Переход `Отменена` — из любого подтверждённого состояния.

#### US-CHG-5 — Корректный запуск приложения из `rent-agent/`

**Заменяет в AS-IS:** `.env` в корне репозитория, `dotenv` ищет файл в `cwd` → при `cd rent-agent && npm start` секреты не загружаются (GAP-001).

**Поток:** разработчик/хостинг запускает агент из `rent-agent/` — все переменные из `rent-agent/.env` (или явный `path` в коде) подхватываются.

#### US-CHG-6 — Двунаправленная синхронизация календарей (Авито / прямая бронь → ЦИАН / Яндекс через RealtyCalendar)

**Заменяет в AS-IS:** `AS_IS.md` §3.5 — «RealtyCalendar: только входящий вебхук, мы не дёргаем их API»; §1 — для брони, появившейся **не** из RealtyCalendar, ничего не блокируется на сторонних агрегаторах. Также снимает ограничение из v1.0 спеки (§6.2 предыдущей версии: «блокировка дат в ЦИАН / Яндекс из агента — out of scope»).

**Контекст.** RealtyCalendar выступает «хабом» для Яндекс Путешествий и ЦИАН: его собственный календарь, обновлённый через API, автоматически проброcит даты в подключённые агрегаторы. Поэтому агенту достаточно одного исходящего вызова RC API на каждое подтверждённое бронирование, появившееся **не** через `POST /webhook/realtycalendar`.

**Актор:** агент (по триггеру), владелец (подтверждает бронь).

**Триггеры подтверждения брони, требующие синхронизации в RC:**

1. **Авито-бронь (через чат).** Гость в Авито согласовал даты, владелец нажимает в Telegram inline-кнопку `Подтвердить бронь` на черновике из US-CHG-3.
2. **Прямая бронь в Telegram.** Гость общается с «Анной», даты согласованы, владелец нажимает `Подтвердить бронь` в Telegram (расширение US-CHG-4).
3. **Booking-вебхук Авито** (если/когда подключён реальный booking-payload от Авито) — обрабатывается аналогично чат-сценарию, но без шага владельца.

**Поток (TO-BE):**

1. Триггер срабатывает → агент берёт запись Notion по `pageId`.
2. Если `Источник` ≠ `Яндекс Аренда`/`ЦИАН` (т.е. бронь пришла **не** через RC) и есть валидные `dateFrom` / `dateTo` — агент вызывает **исходящий API RealtyCalendar** для блокировки дат у объекта.
3. RealtyCalendar пробрасывает блок в ЦИАН и Яндекс по своей синхронизации.
4. Агент обновляет в Notion: `Статус` = `Подтверждена`, отметка `Синхронизировано с RC` (новое булево/чекбокс-поле).
5. Владельцу — короткое подтверждение в Telegram («Даты заблокированы в ЦИАН и Яндексе»).

**Идемпотентность:** повторный триггер по той же брони не должен повторно блокировать те же даты в RC. Контроль — через флаг `Синхронизировано с RC` или внешний ID, возвращаемый RC.

**Обратное направление (для полноты картины):** уже работает в US-OK-1 — RC → Авито (`AS_IS.md` §1 п.1). Меняется только **наименование** в документации: это «прямое направление двунаправленной синхронизации», обратное реализуется этим сценарием.

**Критерий приёмки:**

- Бронь, созданная как черновик Авито (US-CHG-3) и подтверждённая владельцем, в течение ≤ 30 секунд видна как «занято» в Яндексе и ЦИАНе (через RC).
- Повторное нажатие `Подтвердить бронь` не создаёт второй блок и не ломает первый.
- Бронь, пришедшая через `POST /webhook/realtycalendar` (Источник = `Яндекс` / `ЦИАН`), **не** инициирует обратный вызов в RC (защита от петли).

## 3. Функциональные требования

Нумерация только для **новых или изменённых** требований. Уже работающие вебхуки RealtyCalendar / базовый Telegram / базовый Авито — см. `AS_IS.md`, не дублируются.

### Инфраструктура и конфигурация

**FR-1.** При старте из каталога `rent-agent/` (в т.ч. `npm start` с `cwd = rent-agent`) все переменные окружения из `.env` этого каталога должны загружаться.  
*Закрывает GAP-001.*

### Telegram: привязка и диалог

**FR-2.** При входящем `POST /webhook/telegram` с текстом сообщения агент должен пытаться распознать **код брони** (совпадение с полем `ID Брони` в Notion, регистронезависимо, с нормализацией пробелов).

**FR-3.** При входящем сообщении с `message.contact` агент должен искать бронь по полю `Телефон` (нормализация: только цифры, учёт ведущей `7`/`8` для РФ).

**FR-4.** При успешной привязке по FR-2 или FR-3 агент обновляет запись Notion: поле `Telegram chat_id` = `chat_id` отправителя.

**FR-5.** Если бронь найдена по `Telegram chat_id` (после привязки или ранее), `generateReply` вызывается с полным контекстом брони; уведомление «неизвестный гость» **не** отправляется.

**FR-6.** Если бронь не найдена и привязка не удалась, поведение как в AS-IS: ответ в режиме первичной консультации + однократная подсказка гостю («отправьте код брони или номер через кнопку контакта») + уведомление владельцу.

*Закрывает GAP-002, GAP-008 (проактивное приветствие гостю после RealtyCalendar **не** в скоупе — гость инициирует диалог или привязывается по коду/телефону).*

### Notion: поиск и статусы

**FR-7.** Реализовать `findBookingByBookingId(bookingId)` и `findBookingByPhone(phone)` в сервисе Notion.

**FR-8.** `updateBookingStatus(pageId, status)` вызывается из production-пайплайна при обработке `callback_query` от inline-кнопок владельца (FR-10).

**FR-9.** Допустимые переходы статусов согласованы с §2.2 US-CHG-4; недопустимый переход логируется, владельцу — короткое сообщение об ошибке.

### Telegram: владелец

**FR-10.** Уведомление владельцу о новой брони (существующий `formatBookingNotification`) дополняется **inline-клавиатурой** для смены статуса (`callback_data` содержит `pageId` и целевой статус).

**FR-11.** Обработчик `callback_query` в роуте Telegram: валидация, что `from.id` соответствует `TELEGRAM_OWNER_CHAT_ID`, затем FR-8.

*Закрывает GAP-004.*

### Авито: реальная интеграция и CRM

**FR-12.** При первом сообщении в чате Авито (нет записи с данным `Авито chat_id`) агент создаёт черновую бронь в Notion со статусом `Ожидает подтверждения`, источником `Авито`, заполненным `Авито chat_id` и (если приходит в payload) `Авито item_id`.

**FR-13.** Реализовать `findBookingByAvitoChatId(chatId)` в сервисе Notion. При повторных сообщениях в том же чате агент использует его и не создаёт дубликат; при наличии новых данных (имя, телефон, даты) — обновляет существующую запись.

**FR-14.** `generateReply` для Авито вызывается с контекстом из найденной/созданной записи Notion (имя, квартира, даты, статус, источник — что заполнено), а **не** с пустым `{}`.

**FR-18.** Webhook URL агента (`<PUBLIC_URL>/webhook/avito`) регистрируется в кабинете/аккаунте Авито (вручную или одноразовым API-вызовом подписки на события Messenger). Это **инфраструктурное** действие при онбординге — не код агента, но обязательное условие работы US-CHG-3. В `README.md` / runbook должен быть пункт «как прописать вебхук в Авито».

*Закрывает GAP-006, GAP-007 и переводит Авито-канал из demo в реальный production.*

### Двунаправленная синхронизация календарей

**FR-15.** Сервис `services/realtycalendar.js` (новый) реализует исходящий метод `blockDates({ objectId, dateFrom, dateTo, externalRef })` — POST в API RealtyCalendar для блокировки дат у объекта, возвращает ID/ссылку на созданную запись в RC.

**FR-16.** Триггеры обращения к `realtycalendar.blockDates`:

1. Обработчик `callback_query` владельца с `action=confirm_booking` (новая кнопка в уведомлении US-CHG-3 / прямой бронь Telegram).
2. (Опционально, если Авито пришлёт реальный booking-payload) — отдельная ветка `payload.type === 'booking'` в `routes/avito.js`.

При триггере: загрузить запись Notion → проверить, что `Источник` ∉ {`Яндекс Аренда`, `ЦИАН`} (защита от петли с US-OK-1) → проверить отсутствие отметки `Синхронизировано с RC` → вызвать `blockDates` → выставить отметку и обновить `Статус` = `Подтверждена`.

**FR-17.** Идемпотентность: в схему Notion добавляется поле `Синхронизировано с RC` (checkbox). Если значение `true` — повторный вызов FR-16 пропускается, владельцу возвращается сообщение «уже синхронизировано». При ошибке RC API — поле остаётся `false`, владельцу — алерт, повтор по нажатию кнопки безопасен.

*Закрывает разрыв «нет блокировки ЦИАН/Яндекс из агента» (`AS_IS.md` §3.5; v1.0 §6.2).*

### Без изменений (ссылка на AS-IS)

Следующее **остаётся** как в коде сегодня и **не входит** в нумерованные FR выше:

- `POST /webhook/realtycalendar`: только `create_booking`, дедуп по `ID Брони`, блок дат Авито, уведомление владельцу.
- Fire-and-forget `200 OK` на всех вебхуках.
- LLM fallback при ошибке OpenRouter.
- Демо-режим Авито.

## 4. Нефункциональные требования

**NFR-1.** Файл секретов: `rent-agent/.env` (шаблон `rent-agent/.env.example`); корневой `.env` репозитория не является источником истины для рантайма агента. *GAP-001.*

**NFR-2.** Вебхуки возвращают `200 OK` ≤ 100 мс; тяжёлая работа — асинхронно (без изменений паттерна AS-IS).

**NFR-3.** Каждый внешний вызов (Notion, OpenRouter, Telegram, Авито) в отдельном `try/catch`; ошибка одного шага не рвёт остальные шаги pipeline.

**NFR-4.** Секреты только в `.env`; файл в `.gitignore`.

**NFR-5.** Логирование: ISO-время, модуль, ключевые шаги (привязка chat_id, создание черновика Авито, смена статуса).

**NFR-6.** Время ответа гостю (Telegram / Авито) с учётом LLM: целевое ≤ 60 с (BG-5); таймаут OpenRouter 30 с с fallback-текстом.

**NFR-7.** Дедупликация броней RealtyCalendar по `booking_id` сохраняется; для черновиков Авито — дедуп по `Авито chat_id`; для исходящего вызова RC API — дедуп по флагу `Синхронизировано с RC` (FR-17).

**NFR-8.** Защита от петли «RC → Авито → RC»: в `services/realtycalendar.blockDates` агент **не** вызывает RC API для записей с `Источник` ∈ {`Яндекс Аренда`, `ЦИАН`} (они уже пришли из RC).

**NFR-9.** Время от нажатия кнопки `Подтвердить бронь` до отметки «занято» на ЦИАН/Яндексе (через RC) — ≤ 30 секунд (включая сетевую задержку RC и его внутреннюю синхронизацию).

**NFR-10.** `DEMO_MODE` — **только** для локальной разработки и CI-тестов (`tests/e2e_mock.js`). В production-окружении переменная должна быть `false` или отсутствовать; при `DEMO_MODE=true` в production-логах появляется явный warning. При `NODE_ENV=production` и неполных Авито-ключах агент завершается с ошибкой конфигурации, а не уходит в demo-молчание (защита от ситуации «вебхук в Авито прописан, но агент думает, что он в demo»).

## 6. Что НЕ входит в скоп

### 6.1 Явно отложено (решения продакта / GAP)

| Элемент | Причина |
|---------|---------|
| Верификация подписи вебхуков (RealtyCalendar HMAC, `X-Telegram-Bot-Api-Secret-Token`, Авито) | GAP-003: публичный URL ещё не принимает реальные брони; включить при выходе в прод |
| Проактивное сообщение гостю в Telegram сразу после брони из RealtyCalendar | GAP-008: привязка через код/контакт; гость сам пишет боту |
| Автозаполнение поля `Заметки` | GAP-005 — DEFER |
| Замена `node-telegram-bot-api` на axios | GAP-009 — DEFER |
| Агрегация async-ошибок в alert-канал | GAP-015 — DEFER |

### 6.2 Вне продукта (как в v1.0)

- Приём онлайн-оплат и эквайринг.
- Мобильное приложение для владельца.
- Прямая интеграция Booking.com / Суточно.ру (вход только через RealtyCalendar, если настроено).
- Dynamic pricing.
- Договоры / акты гостям.
- Дашборды аналитики и P&L (данные копятся в Notion; отчёты — ручные или отдельный проект).
- **Прямые** интеграции агента с ЦИАН и Яндекс Путешествиями: блокировка идёт **только** через исходящий API RealtyCalendar (US-CHG-6, FR-15). Если RC не подключён к одному из агрегаторов — блокировка туда не доходит, агент это **не** компенсирует.
- Управление несколькими объектами аренды через одну переменную `REALTYCALENDAR_OBJECT_ID`: если квартир несколько, `objectId` обязан храниться в карточке Notion (новое поле — задача в бэклоге, в текущем релизе одна квартира = одна переменная окружения).

### 6.3 Критерий готовности релиза TO-BE

- US-OK-1, 3, 4, 6 регрессионно зелёные (US-OK-2 и US-OK-5 удалены, см. §2.1).
- US-CHG-1…6 и FR-1…FR-18 приняты по чек-листу §2.2.
- Webhook URL прописан в кабинете Авито (FR-18) — проверено боевым сообщением в Messenger.
- В production `DEMO_MODE=false`, реальные Авито-ключи валидны; в логах нет warning'ов NFR-10.
- Двунаправленная синхронизация проверена end-to-end: бронь Яндекс → Авито (US-OK-1) и подтверждённая бронь Авито → ЦИАН/Яндекс (US-CHG-6).
- На каждое реальное входящее сообщение Авито в Notion появляется/обновляется запись (BG-4).
- Метрики BG-1…BG-5 измеряются после выката на публичный URL (отдельный этап).
