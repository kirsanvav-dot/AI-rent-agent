# Универсальная методика создания AI-агента

> Единый документ для всех случаев: с нуля или на существующем коде, через Cursor или ГигаЧат, в IDE или через CLI. Перед каждым разделом указано, кому он нужен — лишнее пропускайте.

---

## Как пользоваться этим документом

Документ построен по принципу **общая база + условные ветки**. В начале каждого раздела стоит маркер:

- 🟢 **ДЛЯ ВСЕХ** — раздел обязательный.
- 🟡 **ТОЛЬКО ЕСЛИ [условие]** — раздел нужен только в определённом случае.
- 🔵 **АДАПТАЦИЯ** — раздел показывает, как изменить шаг под конкретный инструмент.

Сначала пройдите блок "Определите ваш случай" — он скажет, какие разделы вам читать.

## 🟢 Главные принципы (не пропускать никогда)

Эти четыре принципа работают в любом сценарии. Если что-то одно нарушено — методика ломается.

### Принцип 1. Документы-основания важнее инструмента

Минимум три документа в проекте: **что делаем** (SPECIFICATION.md), **как устроено** (ARCHITECTURE.md), **в каком порядке** (TRACKER.md). Они переживают смену LLM, смену чата и смену вас самих.

### Принцип 2. Две модели — две роли

- **Мощная** (Opus, GPT-5, GigaChat Max) — думать, планировать, проектировать.
- **Рабочая лошадка** (Sonnet, GPT-5-mini, GigaChat Lite) — исполнять готовый план.

Использовать одну модель на всё — это либо дорого, либо слабо.

### Принцип 3. Один шаг — один промпт — одна проверка

Не давайте агенту весь проект сразу. По одной задаче с проверкой результата. Как с младшим сотрудником.

### Принцип 4. Изоляция контекста

Для шага N давайте только то, что нужно для шага N. Не `@*` и не "посмотри весь проект".

## 🟡 ТОЛЬКО ДЛЯ BROWNFIELD: Шаг 0а — Археология (AS_IS.md)

> Пропустите этот раздел, если стартуете с нуля.

Прежде чем что-то менять в существующем коде, нужно понять, что уже есть.

**Промпт (модель: мощная, контекст: вся кодовая база):**

```
Выступи в роли агента-археолога. Изучи кодовую базу и напиши AS_IS.md.

Структура:
1. Что система делает сейчас (функции, своими словами).
2. Архитектурная карта: модули и связи.
3. Внешние интеграции и данные, которыми система обменивается.
4. Структура данных (БД, файлы, кэш).
5. Технический долг и "странные места" — где есть риск что-то сломать.
6. Точки расширения — где безопаснее добавлять новое.

ОТДЕЛЬНЫЙ раздел в конце: "## Обнаруженные пробелы (GAPs)"
Сюда вынеси упоминания функционала, который:
- Заявлен в старых README, CHANGELOG, комментариях TODO/FIXME,
  закомментированном коде, или в эндпоинтах, возвращающих "not implemented".
- Найдены отключённые feature flags, выглядящие как незавершённый функционал.
- НО фактически не работает в текущем коде.

Для каждого пробела укажи:
- Описание функции.
- Где найдено упоминание (файл, строка, контекст).
- Что фактически реализовано (если что-то частичное).
- Что выглядит как причина незавершённости (если можно понять из контекста).

Не делай выводов о том, нужна эта функция или нет. Только фиксируй факт.
Если что-то неясно — явно отметь "ТРЕБУЕТ УТОЧНЕНИЯ".
Не выдумывай.
```

**Что проверить:**
- Все известные вам функции упомянуты в основной части.
- Список интеграций совпадает с реальностью.
- Раздел "Обнаруженные пробелы" вынесен отдельно, не смешан с описанием работающего.
- Помечены "ТРЕБУЕТ УТОЧНЕНИЯ" — это вопросы, которые нужно закрыть до следующего шага.

## 🟡 ТОЛЬКО ДЛЯ BROWNFIELD: Шаг 0б — GAP_ANALYSIS.md

> Пропустите, если стартуете с нуля или если в AS_IS не было обнаружено пробелов.

Это **критически важный шаг**, который часто пропускают. Если в AS_IS обнаружены пробелы (заявленный, но не реализованный функционал) — каждый из них требует осознанного решения **до** написания SPECIFICATION (TO-BE).

### Почему нельзя автоматически переносить пробелы в TO-BE

Соблазн "раз обещали — давайте сделаем" — ловушка. Четыре причины:

1. **Возможно, функцию специально не сделали.** Сменились приоритеты, нашлась проблема в подходе, рынок изменился. Решение могло быть осознанным, но не задокументированным.
2. **Требование могло устареть.** То, что обсуждалось 2 года назад, сегодня может быть бессмысленным.
3. **Это раздувает скоуп.** "Сделаем заодно" — классический способ удвоить срок проекта.
4. **Может быть скрытое препятствие.** Возможно, реализация технически невозможна или дорога, и об этом знали.

### Промпт для категоризации

**Модель:** мощная. **Контекст:** `@AS_IS.md`.

```
Изучи раздел "Обнаруженные пробелы (GAPs)" в AS_IS.md.

Напиши GAP_ANALYSIS.md. Для каждого пробела дай:

1. Описание (взять из AS_IS).
2. Где обнаружен (ссылка на AS_IS).
3. Категоризация — одна из трёх:
   - GAP-DEFER: зафиксировать, но не делать сейчас.
     Это функционал, который не нужен для текущей задачи и не блокирует её.
   - GAP-BLOCKING: критичен для запланированных изменений.
     Без него новый функционал не работает или работает неверно.
   - GAP-OBSOLETE: явно устарел или потерял смысл.
     Например, относится к давно удалённой части продукта.
4. Влияние на текущий проект (1-2 предложения).
5. Рекомендованное действие:
   - Для DEFER: создать отдельную задачу в бэклоге, в текущий релиз не брать.
   - Для BLOCKING: либо включить в скоуп с обоснованием, либо изменить план.
   - Для OBSOLETE: запротоколировать удаление упоминаний, закрыть вопрос.
6. Вопрос к продакту (если решение неочевидно): что именно нужно
   уточнить у владельца продукта.

Не выдумывай контекст. Если непонятно — пиши "ТРЕБУЕТ РЕШЕНИЯ".
```

### Что с каждой корзиной делать

**GAP-DEFER (большинство находок):**
- Зафиксировать в GAP_ANALYSIS.md.
- Создать задачу в внешнем трекере (Jira, Linear, Notion) с пометкой "Discovered in audit YYYY-MM-DD".
- В SPECIFICATION.md (TO-BE) **не включать.**
- Двигаться дальше.

**GAP-BLOCKING (требует решения человека):**
- Зафиксировать в GAP_ANALYSIS.md.
- **Не двигаться** к SPECIFICATION до решения продакта.
- Три варианта решения:
  1. **Включить в скоуп.** Расширить TO-BE, оценить дополнительное время.
  2. **Изменить план.** Использовать другой подход (например, email вместо отсутствующего Slack-сервиса).
  3. **Остановить проект.** Если включение слишком дорогое, а альтернатив нет.
- Это решение принимает **человек** (продакт, заказчик), не агент.

**GAP-OBSOLETE:**
- Зафиксировать в GAP_ANALYSIS.md с обоснованием.
- Добавить отдельный шаг в будущий TRACKER: "Очистка устаревших упоминаний" (удалить TODO, мёртвые роуты, неактуальные конфиги).
- В TO-BE **не включать.**

### Когда пробел всё-таки попадает в TO-BE

**Только в одном случае:** категория BLOCKING + решение продакта "включаем в скоуп".

Тогда в SPECIFICATION.md этот функционал получает пометку "originally planned, never implemented; included after audit" — чтобы через год было понятно, откуда он взялся.

**Главное:** пробел проходит через тот же фильтр продуктового решения, что и любое другое требование. Не "автоматически делаем потому что обещали", а "осознанно включаем потому что нужно".

> Пропустите, если стартуете с нуля.

Документ, явно описывающий, что меняется. Главная защита от случайных правок.

```
Выступи в роли системного аналитика.

1. Что добавляется (включая BLOCKING-пробелы, принятые в скоуп)
2. Что изменяется

6. Риски и компенсирующие меры
7. Стратегия миграции данных, если нужна

```

Раздел "Что остаётся как есть" критичен — без него агент будет "оптимизировать" то, что трогать не просили.

**Промпт:**

```
Напиши AGENTS.md по официальному формату Agentic AI Foundation.

Это инструкции для AI-агента, который будет работать с проектом.

Структура (только релевантные разделы):

## Project Overview
1-2 предложения: что это за проект.

## Setup
Команды установки и инициализации.

## Build & Run
Точные команды для локального запуска.

## Testing
Точные команды запуска тестов.
[Для brownfield: как запустить регрессию.]

## Code Style
Конвенции: именование, форматирование, обработка ошибок, структура файлов.

## Important Constraints
Что агенту НЕЛЬЗЯ делать:
- Не удалять миграции БД.
- Не коммитить .env.
- Не оптимизировать код, который явно не указан в задаче.
- [Для brownfield: не трогать модули из CHANGE_REQUEST как "остаётся без изменений".]
```

**🔵 АДАПТАЦИЯ ПОД ИНСТРУМЕНТ:**

- **ТИП 1:** агент сам читает файл при открытии проекта.

## 🟢 Часть 4. TRACKER.md

**Модель:** мощная. **Контекст:** `@SPECIFICATION.md` `@ARCHITECTURE.md` (+ `@CHANGE_REQUEST.md` `@GAP_ANALYSIS.md` для brownfield).

**Базовый промпт:**

```
Выступи в роли тимлида. Напиши TRACKER.md — пошаговый план.

Требования:
- Шаг атомарный: 1 шаг = 1 модуль или 1 интеграция.
- От простого к сложному: сначала каркас, потом сервисы, потом бизнес-логика.
- Для каждого шага:
  * Цель (1 предложение)
  * Задачи (чек-боксы)
  * Файлы, которые будут затронуты
  * Как проверить (что я должен увидеть)
  * Готовый промпт для Cursor
- Первый шаг — каркас проекта.
- Предпоследний — сквозной тест.
- Последний — README.
- У каждого шага статус [backlog].
- В конце — раздел "Что делать, если шаг не получается".
```

**🟡 ДОБАВКА ДЛЯ BROWNFIELD:**

```
Дополнительно:
- Первый шаг — прогон baseline-регрессии.
- Каждый шаг ссылается на пункты CHANGE_REQUEST.md.
- Каждый шаг включает прогон регрессионных тестов как обязательную проверку.
- Минимизируй касания существующего кода.
- Если в GAP_ANALYSIS есть OBSOLETE-пробелы — выдели отдельный шаг
  "Очистка устаревших упоминаний".
- Если в GAP_ANALYSIS есть BLOCKING-пробелы, принятые в скоуп —
  они становятся отдельными шагами с пометкой [from gap analysis].
```

## 🟢 Часть 5. Цикл разработки

Теперь переключаемся на рабочую модель (Sonnet, GigaChat Lite/Pro, GPT-5-mini).

### Базовый универсальный промпт для одного шага

```
Мы выполняем Шаг N из TRACKER.md: "[название]".

Контекст: ARCHITECTURE.md, TRACKER.md, [конкретные файлы из шага]

Задачи (из TRACKER.md):
- [задача 1]
- [задача 2]

Требования:
1. Не трогай код из других шагов.
2. Try/catch на каждый async-вызов.
3. Все секреты — через переменные окружения.
4. Логирование в формате [имя_модуля] событие.
5. Напиши tests/test_[имя_шага].js — простой скрипт без фреймворков.
6. Обнови статус шага в TRACKER.md на [done].

Не переходи к следующему шагу — остановись и жди подтверждения.
```

### 🟡 ДОБАВКА ДЛЯ BROWNFIELD:

К требованиям добавьте:

```
7. Не модифицируй файлы, не указанные явно в этом шаге.
8. Соблюдай существующий стиль кодирования.
9. После реализации запусти регрессионные тесты — все должны быть зелёными.
10. Если регрессия упала — НЕ продолжай. Сообщи, что сломалось, и жди указаний.
```

### 🔵 АДАПТАЦИИ ПОД ИНСТРУМЕНТ

**ТИП 1 (Agentic IDE — Cursor, Claude Code, GigaCode):**
Промпт работает как есть. Можно дополнительно поручать `git commit` и запуск тестов:

```
После теста — сделай git add и git commit -m "Шаг N: [название]".
```

**ТИП 2 (Чат с файлами — ГигаЧат в браузере, Claude.ai):**
Добавьте к требованиям:

```
- Верни ПОЛНЫЙ код файла, готовый к копированию. НЕ присылай диффы.
- В конце укажи команду для запуска теста.
- В конце укажи: "СТАТУС: измени Шаг N в TRACKER.md на [done]" — я обновлю руками.
```

**ТИП 3 (Чистый чат):**
Соберите CONTEXT_PACK вручную:

```
=== ARCHITECTURE.md ===
[содержимое]
=== TRACKER.md ===
[содержимое]
=== src/services/notion.js (если существует) ===
[содержимое]

ЗАДАЧА: [описание шага]
ФОРМАТ ОТВЕТА: цельные блоки кода для каждого файла, готовые к копированию.
В конце — команды для терминала.
```

### Что делать после каждого ответа агента

Независимо от типа инструмента:

1. **Запустите тест.** Упал — отдайте ошибку обратно, попросите починить.
2. **🟡 Для brownfield: запустите регрессию.** Если регрессия красная — откатить и разбираться.
3. **Сделайте git commit.** Один шаг — один коммит. Это страховка.
4. **Посмотрите глазами** — нет ли "лишних" файлов или изменений.
5. **Проверьте статус в трекере.** Если не обновился — поправьте руками.

### Когда начинать новый чат (для ТИПОВ 2 и 3)

Признаки замусоренного контекста:
- Больше 30-40 сообщений в одной сессии.
- Модель забывает базовые требования (перестала писать try/catch).
- Появились галлюцинации.

**Что делать:** новый чат, заново прикрепить SPEC, ARCH, AGENTS, TRACKER и нужные для следующего шага файлы.

## 🟢 Часть 6. Сквозной тест

После того, как все шаги в трекере отмечены `[done]`:

**Модель:** снова мощная. **Контекст:** `@ARCHITECTURE.md` `@TRACKER.md` `@src/`.

```
Напиши сквозной (e2e) тест всего пайплайна с моками внешних сервисов.

Контекст: ARCHITECTURE.md, SPECIFICATION.md, исходный код проекта.

Проверь все сценарии из SPECIFICATION.md.
Особо проверь:
- Внешний сервис не отвечает.
- Пустой или невалидный вход.
- Таймаут.

[Для brownfield добавь: прогон всех регрессионных тестов — должны быть зелёными.]

Выведи отчёт: сколько кейсов прошло / упало.
```

После сквозного теста — обязательный **ручной приёмочный тест:** 2-3 реальных сценария на реальных данных своими руками.

## 🟢 Часть 7. Запуск

### 🟡 ДЛЯ BROWNFIELD: стратегия безопасного развёртывания

В brownfield нельзя просто "выкатить". Три безопасных подхода:

1. **Feature flag** (рекомендуется новичкам) — новый функционал прячется за переключателем. Включаете для себя, потом для 10%, потом для всех.
2. **Параллельная работа** — старый и новый код работают одновременно, результаты сравниваются.
3. **Strangler Fig** — модули заменяются по одному, старая система постепенно вытесняется.

В любом подходе обязателен **план отката** — что делать при критическом баге в проде.

### Хостинг

Самые простые варианты для непрограммиста — Railway или Render. Подключите GitHub-репозиторий, пропишите переменные окружения, получите публичный URL.

```
Напиши README.md с инструкцией по запуску для непрограммиста:
1. Как создать аккаунт на Railway / Render.
2. Как подключить GitHub.
3. Список переменных окружения с пояснениями.
4. Как проверить, что агент запустился.
```

## 🟢 Главные ошибки и как их избежать

| Ошибка | Последствие | Защита |
|---|---|---|
| Пропустить SPECIFICATION | Через 2 дня все забыли цель | Всегда начинать со спеки |
| Одна модель на всё | Дорого или слабые архитектурные решения | Мощная — план, рабочая — исполнение |
| Дать агенту весь проект через `@*` | Расход токенов, поломка несвязанного | Изолируйте контекст |
| Не делать git commit | Не откатиться, если сломалось | Один шаг — один коммит |
| 🟡 Автоматически перенести все gap в TO-BE | Скоуп удваивается, проект провисает | GAP_ANALYSIS с решением по каждому пробелу |
| 🟡 Игнорировать находки в AS_IS | Потеря важных сигналов, повторение находок через год | Фиксировать всё в GAP_ANALYSIS, даже DEFER |
| 🟡 Команда "оптимизируй код" в brownfield | Агент перепишет половину проекта | Все улучшения через CHANGE_REQUEST |
| 🟡 Игнорировать "странный" код в brownfield | Сломаете кейс, о котором забыли | Сначала разберитесь зачем, потом меняйте |
| Слишком длинный AGENTS.md | Снижение качества работы агента | Держите файл коротким, до 80 строк |
| Использовать AGENTS.md как замену SPEC | Агент не понимает бизнес-цель | AGENTS — про "как работать", SPEC — про "что делаем" |

## 🟢 Резюме: ваш процесс на одной странице

**Принцип:** SDD + GAP-обработка + AGENTS.md = полный контроль над AI-разработкой.

**Документы-роли:**

- **AS_IS** — что есть сейчас (только brownfield).
- **GAP_ANALYSIS** — что было обещано, но не сделано: что с этим делать (только brownfield).
- **SPECIFICATION** — что строим (бизнес).
- **ARCHITECTURE** — как устроено (техника).
- **CHANGE_REQUEST** — что меняем (только brownfield).
- **AGENTS.md** — как с проектом работать (для агента).
- **TRACKER** — в каком порядке делаем.

**Поток:** Goal Alignment → (AS_IS → GAP_ANALYSIS → решение продакта →) SPECIFICATION → ARCHITECTURE → (CHANGE_REQUEST →) AGENTS.md → (регрессия baseline →) TRACKER → Цикл разработки → Сквозной тест → Запуск.

**Принципы:**

1. Мощная модель планирует. Рабочая исполняет.
2. Один шаг — один промпт — одна проверка — один коммит.
3. Изолируйте контекст: только нужное для текущего шага.
4. Пробел из AS_IS — это сигнал, а не задача. Решение по каждому принимает человек.

Принципы остаются одинаковыми, что бы вы ни использовали — Cursor с Claude, GigaCode с ГигаЧатом, или чистый чат через копипаст. Меняется только техника передачи контекста.
