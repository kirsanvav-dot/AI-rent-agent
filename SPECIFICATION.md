## 1. Бизнес-контекст и цели

### 1.1 Проблема

Владелец квартиры для посуточной аренды вынужден:

- Вручную отвечать клиентам в Telegram, Авито и других каналах (1–2 часа 3 раза в неделю).
- Синхронизировать календари вручную: бронь с одной площадки нередко конфликтует с другой
- Платить комиссию агрегаторам (Booking.com, Суточно.ру) за каждую бронь
- Терять данные о гостях из-за отсутвия времени ведения аналитики 
- Терять данные о чистой прибыли с квартиры из-за отсутвия времени ведения аналитики

### 1.2 Решение

Единый **AI-агент** на базе Node.js / Express, который:

1. Принимает вебхуки от всех площадок в одной точке входа.
2. Автоматически создаёт и обновляет брони в Notion-CRM.
3. Синхронизирует доступность дат на Авито.
4. Общается с клиентами в Telegram от имени «Анны» (LLM-персонаж) и в чатах Авито.
5. Уведомляет владельца о новых бронях и важных событиях.

### 1.3 Бизнес-цели


| #    | Цель                                  | Метрика успеха                                                                  |
| ---- | ------------------------------------- | ------------------------------------------------------------------------------- |
| BG-1 | Сократить время ручной работы на 80 % | ≤ 10 мин/день ручного труда к концу 1-го месяца                                 |
| BG-2 | Устранить двойные бронирования        | 0 конфликтов дат в течение 30 дней                                              |
| BG-3 | Снизить зависимость от агрегаторов    | Доля прямых броней (Авито /Яндекс Путешествия/ЦИАН/ Telegram) ≥ 60 % за квартал |
| BG-4 | Иметь единую CRM для всех броней      | 100 % броней записаны в Notion в режиме реального времени                       |
| BG-5 | Отвечать клиентам быстрее             | Время первого ответа ≤ 60 секунд 24/7                                           |

## 3. Пользовательские истории: AS-IS vs TO-BE

### 3.1 Поток «Новая бронь с агрегатора (Яндекс, ЦИАН)»

#### AS-IS (сейчас, вручную)

```
Гость бронирует квартиру на Яндекс Путешествии
  → владелец получает push-уведомление
  → вручную открывает RealtyCalendar
  → проверяет корректность дат и суммы
  → RealtyCalendar блокирует даты на Авито (иначе двойное бронирование)
  → вручную вносит данные в таблицу Google Sheets
  → пишет гостю в Telegram: «Добрый день! Ваша бронь подтверждена...»
Время: 15–30 минут, риск ошибки: высокий
```

#### TO-BE (с агентом)

```
Гость бронирует квартиру на Яндекс Путешествии
  → RealtyCalendar отправляет вебхук POST /webhook/realtycalendar
  → Агент создаёт запись в Notion CRM (< 2 сек)
  → Агент блокирует даты на Авито через STR API (< 5 сек)
  → Агент отправляет уведомление владельцу в Telegram (< 2 сек)
Время: < 10 секунд, риск ошибки: минимальный
```

---

### 3.2 Поток «Гость пишет в Telegram»

#### AS-IS

```
Гость: «Добрый день, можно узнать адрес?»
  → владелец получает уведомление на телефон
  → открывает переписку
  → вспоминает / ищет, кто этот гость и когда заезд
  → пишет ответ вручную
Время ответа: 5 минут – несколько часов
```

#### TO-BE

```
Гость: «Добрый день, можно узнать адрес?»
  → Агент получает POST /webhook/telegram
  → Ищет бронь гостя в Notion по chat_id
  → Передаёт контекст (даты, имя) в OpenRouter LLM
  → LLM генерирует ответ от имени «Анны» с нужными деталями
  → Гость получает ответ за ≤ 60 секунд 24/7
Владелец уведомляется, только если агент не смог ответить
```

---

### 3.3 Поток «Гость пишет в чате Авито»

#### AS-IS

```
Гость в чате Авито: «Свободно на 20-22 июня?»
  → владелец открывает приложение Авито или RealtyCalendar
  → проверяет календарь
  → отвечает вручную
Время ответа: 10 минут – несколько часов, конкуренты отвечают быстрее
```

#### TO-BE

```
Гость в чате Авито: «Свободно на 20-22 июня?»
  → Авито отправляет вебхук POST /webhook/avito (event: message)
  → Агент проверяет, что автор ≠ владелец
  → Агент проверяет, что квартира свободна
  → Генерирует ответ через LLM (с пустым или минимальным контекстом)
  → Отправляет ответ в чат Авито через Messenger API
Время ответа: < 60 секунд
```

---

### 3.4 Поток «Владелец хочет видеть все брони»

#### AS-IS

```
Владелец смотрит в несколько источников: icloud календарь, RealtyCalendar
Риск: дублирование, пропуск информации
```

#### TO-BE

```
Владелец открывает Notion CRM
  → Видит все брони (из всех источников) в единой таблице
  → Фильтрует по статусу, датам
  → Может позвонить / написать гостю прямо из карточки
```

## 4. Функциональные требования

### FR-1: Вебхук RealtyCalendar


| Атрибут                          | Значение                                                                                                                                                                                                      |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Метод и путь                     | `POST /webhook/realtycalendar`                                                                                                                                                                                |
| Триггер                          | `action === "create_booking"`                                                                                                                                                                                 |
| Обязательные поля входящего JSON | `booking_id`, `date_from`, `date_to`, `total_price`, `guest.name`, `guest.phone`, `property.title`, `booking_origin.title`                                                                                    |
| Действия агента                  | 1. Ответить `200 OK` немедленно. 2. Асинхронно: создать бронь в Notion → заблокировать даты на Авито -> заблокировать даты через RealtyCalendar в ЦИАН и Яндекс Путешествии → уведомить владельца в Telegram. |
| Обработка ошибок                 | Каждый шаг в отдельном try/catch; ошибки логируются, не останавливают pipeline                                                                                                                                |


### FR-2: Вебхук Telegram


| Атрибут           | Значение                                                                            |
| ----------------- | ----------------------------------------------------------------------------------- |
| Метод и путь      | `POST /webhook/telegram`                                                            |
| Условие обработки | Только `message` апдейты (не `edited_message`, не `callback_query`)                 |
| Логика            | `chat_id` → `findBookingByChatId` → `generateReply(text, context)` → `sendMessage`  |
| Fallback          | Если бронь не найдена: генерировать ответ с пустым контекстом + уведомить владельца |
| Ответ             | `200 OK` до завершения обработки (async fire-and-forget)                            |


### FR-3: Вебхук Авито


| Атрибут      | Значение                                                                  |
| ------------ | ------------------------------------------------------------------------- |
| Метод и путь | `POST /webhook/avito`                                                     |
| Триггер      | `payload.type === "message"` и автор ≠ `AVITO_USER_ID`                    |
| Логика       | Получить `access_token` → `generateReply(text, {})` → `avito.sendMessage` |
| Ответ        | `200 OK` немедленно                                                       |


### FR-4: Notion CRM

- Создание записи о брони (функция `createBooking`).
- Поиск брони по Telegram `chat_id` (функция `findBookingByChatId`).
- Опциональное обновление статуса брони (функция `updateBookingStatus`).

### FR-5: LLM (OpenRouter)

- Системный промпт с персонажем «Анна — вежливый менеджер по аренде».
- В системный промпт подставляется контекст брони: квартира, даты, имя гостя.
- Если контекст пуст — агент отвечает в режиме «уточняю детали».
- Модель по умолчанию: `openai/gpt-4o-mini` (настраивается через `OPENROUTER_MODEL`).

## 6. Требования к интеграциям

### 6.1 Notion API

- Библиотека: `@notionhq/client`
- Аутентификация: `NOTION_TOKEN` (Integration Token)
- Целевая база данных: `NOTION_DATABASE_ID`
- API-версия: текущая стабильная (`2022-06-28`)

**Необходимые разрешения интеграции Notion:**

- Read content
- Insert content
- Update content

### 6.2 OpenRouter API

- HTTP-клиент: `axios`
- Endpoint: `POST https://openrouter.ai/api/v1/chat/completions`
- Обязательные заголовки: `Authorization: Bearer <key>`, `HTTP-Referer`, `X-Title`
- Ключ: `OPENROUTER_API_KEY`
- Таймаут запроса: 30 сек

### 6.3 Telegram Bot API

- Библиотека: `node-telegram-bot-api` (режим `polling: false` — только вебхуки)
- Вебхук регистрируется вручную через:  
`https://api.telegram.org/bot<TOKEN>/setWebhook?url=<YOUR_URL>/webhook/telegram`
- Токен: `TELEGRAM_BOT_TOKEN`
- chat_id владельца: `TELEGRAM_OWNER_CHAT_ID`
- Формат уведомлений владельцу: HTML parse mode

### 6.4 RealtyCalendar

- Входящий вебхук (push от RealtyCalendar → наш сервер)
- Метод: `POST /webhook/realtycalendar`
- Опциональная верификация подписи: `REALTYCALENDAR_WEBHOOK_SECRET` (HMAC-SHA256)
- Обязательный ответ: `200 OK` в течение 3 секунд (иначе RealtyCalendar повторит запрос)

### 6.5 Авито STR API

- HTTP-клиент: `axios`
- Base URL: `https://api.avito.ru`
- Аутентификация: OAuth2 Client Credentials  
`POST https://api.avito.ru/token` (form-data: `client_id`, `client_secret`, `grant_type=client_credentials`)
- Токен кешируется в памяти и обновляется до истечения срока
- Ключи: `AVITO_CLIENT_ID`, `AVITO_CLIENT_SECRET`, `AVITO_USER_ID`

**Используемые эндпоинты Авито:**


| Действие            | Метод | Путь                                                        |
| ------------------- | ----- | ----------------------------------------------------------- |
| Получить токен      | POST  | `/token`                                                    |
| Заблокировать даты  | POST  | `/core/v1/accounts/{user_id}/items/{item_id}/bookings`      |
| Отправить сообщение | POST  | `/messenger/v1/accounts/{user_id}/chats/{chat_id}/messages` |

## 7. Структура данных: База данных Notion

### 7.1 База «Брони» (Bookings)

Каждая строка = одна бронь.


| Поле Notion        | Тип          | Обязательное | Описание                                                   |
| ------------------ | ------------ | ------------ | ---------------------------------------------------------- |
| `ID Брони`         | title        | ✅            | Уникальный ID из источника (напр., `RC-12345`)             |
| `Имя клиента`      | rich_text    | ✅            | Полное имя гостя                                           |
| `Телефон`          | phone_number | ✅            | Контактный номер                                           |
| `Даты`             | date         | ✅            | Дата заезда (`start`) и выезда (`end`)                     |
| `Квартира`         | rich_text    | ✅            | Название объекта (напр., «1й проезд Марьиной рощи д11»)    |
| `Источник`         | select       | ✅            | Площадка: `Авито`, `Яндекс`, `ЦИАН`, `Прямое бронирование` |
| `Сумма`            | number       | ✅            | Итоговая сумма брони в рублях                              |
| `Статус`           | select       | ✅            | `Подтверждена`, `Ожидает`, `Отменена`, `Завершена`         |
| `Telegram chat_id` | number       | —            | chat_id гостя для связи через Telegram                     |
| `Авито item_id`    | rich_text    | —            | ID объявления на Авито для блокировки дат                  |
| `Авито chat_id`    | rich_text    | —            | ID чата в Авито Messenger                                  |
| `Заметки`          | rich_text    | —            | Дополнительные комментарии менеджера / агента              |
| `Дата создания`    | created_time | —            | Заполняется Notion автоматически                           |


### 7.2 Допустимые значения select-полей

**Источник:**

- Авито
- Яндекс Аренда
- ЦИАН
- Telegram (прямое)
- Другое

**Статус:**

- Ожидает подтверждения
- Подтверждена
- Активна (гость заселён)
- Завершена
- Отменена

### 7.3 Пример объекта `BookingData` в коде

```js

  bookingId: 'RC-12345',            // string — ID Брони (title)
  guestName: 'Иванов Иван',         // string — Имя клиента
  phone: '+79001234567',            // string — Телефон
  dateFrom: '2026-06-20',           // ISO date — начало аренды
  dateTo: '2026-06-22',             // ISO date — конец аренды
  apartment: '1й проезд Марьиной рощи д11',    // string — Квартира (select)
  source: 'Яндекс Аренда',          // string — Источник (select)
  totalPrice: 7500,                 // number — Сумма (руб.)
  telegramChatId: null,             // number | null
  avitoItemId: null,                // string | null
  avitoChatId: null,                // string | null
  notes: '',                        // string
};
```

## 8. Контекст для LLM: структура объекта `context`

```js
// Передаётся в llm.generateReply(userMessage, context)
const context = {
  guestName: 'Иван',                // имя гостя для обращения
  apartment: '1й проезд Марьиной рощи д11',    // объект аренды
  dateFrom: '20 июня 2026',         // дата заезда (локализованная)
  dateTo: '22 июня 2026',           // дата выезда
  totalPrice: 7500,                 // сумма брони
  source: 'Яндекс Аренда',          // откуда пришёл гость
};

// Если бронь не найдена, передаётся пустой объект: {}
```

**Системный промпт агента «Анна»:**

```
Ты — Анна, вежливый и дружелюбный менеджер по посуточной аренде квартир.
Отвечай коротко (2–4 предложения), тепло и по делу.
Если знаешь данные гостя — используй их (имя, квартира, даты).
Если не знаешь ответа на вопрос — напиши: «Уточню для вас и отвечу в ближайшее время 😊».
Не придумывай адреса, цены и условия, если они не указаны в контексте.
```

## 10. Ограничения и риски


| Риск                              | Вероятность | Митигация                                                                         |

| Превышение лимитов OpenRouter     | Низкая      | `try/catch` + fallback-сообщение «Анна сейчас недоступна, ответим в течение часа» |
